# Руководство по CliKit

Это подробное руководство по использованию CliKit, плагина для OpenCode, который превращает сырой запрос в упорядоченный рабочий процесс разработки.

## 1. Введение

CliKit — это слой рабочего процесса для OpenCode, предоставляющий набор специализированных агентов, команд, хуков и системы памяти для управления процессом разработки от обсуждения до финальной поставки. Он помогает структурировать процесс, обеспечивая согласованность между планированием, выполнением, проверкой и передачей работы.

**Зачем нужен CliKit?**

OpenCode — мощный инструмент, но поддержание порядка в больших задачах требует дисциплины. CliKit добавляет эту структуру, предоставляя:

- **Оркестрацию рабочих процессов**: Команды для обсуждения, создания, запуска, проверки и поставки изменений.
- **Библиотеку агентов**: Специализированные роли для сборки, планирования, исследования, ревью, поиска информации, анализа и визуализации.
- **Систему памяти**: Артефакты обсуждений, планов, исследований, ревью и передач, которые сохраняются между сессиями.
- **Хуки безопасности и качества**: Защита Git, проверки безопасности, усечение вывода, сводка памяти и вспомогательные инструменты для контекста.
- **Цикл проверки**: Выполнение сжатых пакетов задач с явной проверкой перед внесением изменений.

## 2. Быстрый старт

Установка CliKit состоит из трех основных шагов:

1.  **Установка Beads Rust (`br`)**:
    CliKit использует `beads_rust` (`br`) для отслеживания задач в `.beads/`.
    - **macOS / Linux / Windows (WSL)**:
      ```bash
      curl -fsSL "https://raw.githubusercontent.com/Dicklesworthstone/beads_rust/main/install.sh" | bash
      ```
    - **Windows (нативная сборка)**: Скачайте `br-v*-windows_amd64.zip` с GitHub Releases и добавьте `br.exe` в ваш `PATH`.
    - Проверьте установку:
      ```bash
      br --version
      ```

2.  **Инициализация состояния трекера (`br init`)**:
    В корне вашего проекта выполните:

    ```bash
    cd your-project
    br init
    ```

    Это создаст директорию `.beads/` для отслеживания задач. Если `.beads/` уже существует, используйте `br sync --import-only` после pull или rebase.

3.  **Установка CliKit**:
    Установите плагин глобально для OpenCode:
    ```bash
    bun x clikit-plugin install
    ```
    После установки перезапустите OpenCode. CliKit автоматически добавит необходимые записи MCP серверов (Context7, Grep, Human MCP), если они отсутствуют.

## 3. Рабочие процессы (Workflows)

CliKit предлагает несколько рабочих процессов, адаптированных к сложности задачи. Основной пайплайн выглядит так:

`/discuss → /create → /start → /verify → /ship`

- **`/discuss`**: Фаза предварительного обсуждения для уточнения намерений, подтверждения предположений и сохранения артефакта обсуждения.
- **`/create`**: Исследует кодовую базу, использует контекст обсуждения, выполняет обязательный этап исследования перед планированием и создает структурированный план в формате XML.
- **`/start`**: Выполняет план пакетами задач (Task Packets). Если план отсутствует, создает минимальный встроенный план.
- **`/verify`**: Запускает все 4 этапа проверки (typecheck, тесты, lint, сборка) и глубокий обзор кода. Может использоваться в любое время, не только перед поставкой.
- **`/ship`**: Финализирует работу в общем рабочем пространстве — коммитит, синхронизирует и применяет изменения к основной ветке репозитория. Команда `/pr` используется опционально для исключительных случаев, требующих отдельной ветки для ревью.

**Режимы выполнения:**

- **Быстрый режим (Quick mode)**: Для простых функций или исправлений.
  ```text
  /discuss → /create → /start → /verify → /ship
  ```
- **Глубокий режим (Deep mode)**: Для сложных функций, исследований или изменений, связанных с UI.

  ```text
  /discuss → /create → /design → /start → /verify → /ship
  ```

  Команда `/design` используется для реализации UI/UX с исследованием вариантов и проверкой доступности (a11y).

- **`/research`**: Автономный этап исследования. Читает контекст обсуждения, собирает внешние доказательства и сохраняет отчет, готовый для планирования.
- **`/verify`**: Может использоваться отдельно для аудита существующей работы перед ее применением.

**Примечание**: Все команды работают автономно, но пайплайн рекомендуется для структурированного подхода. `br` и `.beads/` являются предпочтительным источником истины для отслеживания задач.

## 4. Основные команды

CliKit предоставляет 14 команд, доступных через `/command-name`. Они сгруппированы по назначению:

### Рабочие процессы (Workflows)

| Команда     | Автономность | Описание                                                                                                                                  |
| ----------- | ------------ | ----------------------------------------------------------------------------------------------------------------------------------------- |
| `/discuss`  |              | Начинает обсуждение, фиксирует намерения, подтверждает предположения и сохраняет артефакт для планирования.                               |
| `/create`   |              | Исследует кодовую базу, загружает контекст обсуждения, проводит исследование перед планированием и создает структурированный план (XML).  |
| `/research` |              | Проводит автономное исследование, собирает внешние данные и сохраняет отчет для планирования.                                             |
| `/design`   |              | Реализует UI/UX, исследует варианты, проверяет доступность (a11y) с использованием агента Vision.                                         |
| `/start`    |              | Выполняет план по пакетам задач (Task Packets). Создает минимальный план, если он отсутствует.                                            |
| `/verify`   |              | Запускает полную проверку: typecheck, тесты, lint, сборка, а также глубокий обзор кода.                                                   |
| `/ship`     |              | Финализирует изменения: коммит, синхронизация и применение к основной ветке репозитория. Рекомендуется использовать `/verify` перед этим. |
| `/pr`       |              | Опциональный процесс создания Pull Request для исключительных случаев, требующих отдельной ветки для ревью.                               |

### Утилиты

| Команда    | Описание                                                                                                     |
| ---------- | ------------------------------------------------------------------------------------------------------------ |
| `/debug`   | Воспроизводит ошибку, находит коренную причину (5 Whys), исправляет и выполняет регрессионное тестирование.  |
| `/status`  | Предоставляет снимок состояния рабочей среды: состояние трекера задач, Git и активных артефактов.            |
| `/init`    | Инициализирует CliKit: создает необходимые директории и пишет настроенный `AGENTS.md`.                       |
| `/handoff` | Автоматически сохраняет состояние сессии для безопасной паузы и последующего возобновления.                  |
| `/resume`  | Возобновляет работу с последней точки сохранения (`handoff`), без необходимости повторного ввода контекста.  |
| `/commit`  | Автоматически определяет тип и область коммита, создавая идеальное сообщение в формате Conventional Commits. |

## 5. Агенты

CliKit использует 7 специализированных агентов для выполнения различных задач:

| Агент      | Режим     | Описание                                                                                                                        |
| ---------- | --------- | ------------------------------------------------------------------------------------------------------------------------------- |
| `build`    | основной  | Основной исполнитель кода, реализует планы.                                                                                     |
| `plan`     | основной  | Обрабатывает блокировку намерений (`/discuss`) и планирование (`/create`), создавая артефакты обсуждений и планы в формате XML. |
| `oracle`   | под-агент | Глубокий анализ кода и анализ компромиссов архитектуры.                                                                         |
| `research` | под-агент | Специалист по внешним данным; пишет артефакты исследований для `/research` и для этапа перед планированием в `/create`.         |
| `explore`  | под-агент | Быстрое исследование кодовой базы.                                                                                              |
| `review`   | под-агент | Контроль качества кода и ревью.                                                                                                 |
| `vision`   | под-агент | Направление дизайна и визуальная реализация (UI/UX).                                                                            |

В сжатом рабочем процессе (`compressed workflow`) по умолчанию активны роли: `build`, `plan`, `review`, а также логика координатора. Агенты `explore`, `research`, `oracle` и `vision` вызываются по требованию.

## 6. Настройка

Конфигурация CliKit осуществляется через файлы `clikit.jsonc` (предпочтительно) или `clikit.json`. Файл может располагаться в одном из следующих мест:

- **Пользовательский (глобальный)**:
  - Linux/macOS: `~/.config/opencode/clikit.jsonc`
  - Windows: `%APPDATA%\opencode\clikit.jsonc`
- **Проектный**: `.opencode/clikit.jsonc` в корне проекта.

Конфигурация проекта имеет приоритет над пользовательской.

**Пример конфигурации (`clikit.jsonc`)**:

```json
{
  "$schema": "https://unpkg.com/clikit-plugin@latest/schema.json",
  "disabled_agents": ["review"],
  "disabled_commands": ["security"],
  "disabled_skills": ["playwright"],
  "skills": {
    "enable": ["test-driven-development", "systematic-debugging"],
    "disable": ["sharing-skills"]
  },
  "agents": {
    "vision": {
      "model": "openai/gpt-4o"
    },
    "build": {
      "model": "anthropic/claude-sonnet-4-5-20250514",
      "temperature": 0.2,
      "permission": {
        "doom_loop": "deny",
        "external_directory": "ask"
      }
    }
  },
  "hooks": {
    "session_logging": false,
    "tool_logging": false,
    "todo_enforcer": {
      "enabled": true,
      "beads_authoritative": true,
      "warn_on_incomplete": false
    },
    "empty_message_sanitizer": { "enabled": true },
    "git_guard": { "enabled": true },
    "security_check": { "enabled": true },
    "subagent_question_blocker": { "enabled": true },
    "truncator": { "enabled": true, "packet_friendly": true },
    "memory_digest": { "enabled": true, "compact_mode": true },
    "todo_beads_sync": { "enabled": false, "mode": "disabled" },
    "cass_memory": { "enabled": true },
    "beads_context": { "enabled": true, "active_only": true, "ready_limit": 3 }
  },
  "workflow": {
    "mode": "compressed",
    "active_roles": ["build", "plan", "review", "coordinator"],
    "use_packets": true,
    "embed_verify_in_start": true,
    "verify_is_audit": true,
    "subagent_call_budget": 2
  }
}
```

**Основные опции конфигурации**:

- `disabled_agents`, `disabled_commands`, `disabled_skills`: Списки для отключения определенных компонентов.
- `agents`, `commands`, `skills`: Объекты для переопределения настроек конкретных агентов, команд или навыков (например, выбор модели, температуры).
- `hooks`: Настройка различных хуков выполнения (логирование, проверка Git, синхронизация задач и т.д.).
- `workflow`: Настройка рабочего процесса, включая режим (`compressed` или `classic`), активные роли и параметры выполнения пакетов задач.

## 7. Советы

- **Task Packets**: Рабочий процесс CliKit разбивает задачи на небольшие, управляемые "пакеты задач" (Task Packets). Каждый пакет фокусируется на одной задаче, затрагивает 1-3 файла и имеет один набор проверок. Это упрощает выполнение и проверку.
- **Beads (`br`)**: Используйте `br` для отслеживания задач. Он обеспечивает постоянное хранение состояния задач, что особенно полезно при работе в команде или при длительных сессиях.
- **Проверка качества**: Команда `/verify` запускает комплексную проверку, включая тесты, линтинг и сборку. Используйте ее регулярно, чтобы убедиться в качестве кода перед его интеграцией. Команда `/ship` включает самопроверку, но явный вызов `/verify` перед `/ship` повышает уверенность.
- **Память CliKit**: Система памяти сохраняет артефакты (обсуждения, планы, исследования, ревью, хэндоффы), что позволяет восстанавливать контекст между сессиями и обеспечивает преемственность работы. Используйте команды `/handoff` и `/resume` для управления состоянием сессии.
- **AGENTS.md**: Этот файл содержит политику выполнения для всех AI-агентов в репозитории и рабочий процесс трекера. Он является важным документом для понимания того, как агенты взаимодействуют и как управляются задачи.
